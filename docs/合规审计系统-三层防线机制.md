## 二、三层防线详解

### L1 — SKILL.md 预防层

**触发时机**: docx skill 加载时（用户说"做 Word"）

**位置**: `C:\Users\wuyu\.claude\skills\docx\SKILL.md` (L9-L19)

**机制**: 在 skill 正文的第一屏插入强制检查步骤：

```markdown
## ⚠️ PRE-GENERATION CHECK (MANDATORY — DO NOT SKIP)

**Before writing any document generation code**, you MUST:
1. Read memory/word_templates_choice.md
2. Ask the user: "用哪套模板？1. 默认 2. Kami 风格"
3. Wait for user confirmation before proceeding
```

**优势**: 
- 零 token 额外开销
- 在"写代码"这个动作发生之前介入
- 属于 skill 内容本身，随 skill 加载自动激活

**局限**: 
- 依赖 Claude 读到并遵守这段指令（软约束）  
- 如果上下文过长，注意力可能被其他内容分散

---

### L2 — Stop hook 实时拦截层

**触发时机**: 每次 Claude 响应结束（Stop 事件）

**位置**: `C:\Users\wuyu\.claude\scripts\hooks\stop-doc-template-check.js`

**注册**: `C:\Users\wuyu\.claude\settings.json` → `hooks.Stop[0]`

**机制**: 两阶段过滤

```
Stage 1 (bash grep, 零 token):
  → 扫描 transcript 最近 500 行 / 200KB
  → 匹配 20+ 文档生成模式（doc.save / python-docx / docx-js / .pptx / .xlsx 等）
  → 95% 的响应无文档生成 → 直接放行（~10ms CPU）

Stage 2 (模式匹配，零 LLM):
  → 检测到文档生成后，搜索模板讨论合规模式
  → 合规 → 静默放行
  → 违规 → 注入 systemMessage: "[文档模板审计] 检测到生成了 Word 文档..."
```

**支持的文档类型**:
- `.docx` (python-docx / docx-js)
- `.pptx` (python-pptx)
- `.xlsx` (openpyxl)
- 中文上下文检测 ("Word 文档", "生成 Word", etc.)

**检测的合规标记**:
- `用哪套模板` / `模板选择` / `模板偏好`
- `Kami 风格` / `Kami 模板`
- `word_templates_choice` / `word.templates.choice`
- `template preference`

**输出格式**: `{"continue":true,"systemMessage":"[文档模板审计] ..."}`

**优势**:
- 零 token 开销（纯 Node.js grep，无 LLM 调用）
- 95% 的响应不受影响
- 作为 L1 的兜底——如果 Claude 在 skill 加载时跳过了检查，这一层会在交出去之前拦截

**局限**:
- 仅扫描 transcript 尾部（最近 500 行 / 200KB）  
- 跨 turn 匹配不支持——如果模板讨论在前几轮、文档生成在当前轮，可能误报  
- 被设计为非破坏性提醒（不 block，只 warn）

---

### L3 — SessionStart 趋势预警层

**触发时机**: 每次会话启动

**位置**: `C:\Users\wuyu\.claude\scripts\hooks\session-start-compliance-audit.js`

**注册**: `C:\Users\wuyu\.claude\settings.json` → `hooks.SessionStart[0]`

**机制**: 读历史数据 → 算合规率 → 低于阈值则注入上下文

```
1. 扫描所有项目的 observations.jsonl（ECC homunculus 目录）
2. 按 session 分组，计算每个规则的合规率
3. 如果合规率 < 85% 且数据量 ≥ 3 个 session → 注入 additionalContext
4. 否则 → 静默
```

**输出示例** (合规率低于阈值时):

```
[合规审计] 基于过去 14 天的会话数据:

⚠️  Word 模板确认: 合规率 40%（2/5 次触发）
   规则: 生成 Word/PPT/Excel 文档前应先询问模板偏好（默认 vs Kami 风格）
   记忆文件: memory/word_templates_choice.md

建议: 在本次会话中，涉及上述操作时主动确认合规步骤。
```

**优势**:
- 提供历史趋势视角——不只是"这次漏没漏"，而是"最近两周漏了多少次"
- 纯 Node.js，零 LLM 调用
- 数据驱动决策：在会话启动时就把"你需要特别注意"的事情告诉 Claude

**局限**:
- 依赖 observations.jsonl 积累——刚部署时无历史数据，需要几周才能生效
- 阈值 (85%) 是硬编码的，目前不支持动态调整

## 四、数据流（闭环）

```
Step 1: observe.sh (PreToolUse + PostToolUse)
  → 记录 { tool_name, tool_input, tool_output, session_id, timestamp }
  → 追加到 observations.jsonl
  → 每 20 次触发通知 observer 进程

Step 2: observations.jsonl
  → 持久化在 ~/ECC/.claude/homunculus/projects/<id>/
  → 多项目隔离，每个项目独立存储

Step 3: session-start-compliance-audit.js (SessionStart)
  → 读取所有项目的 observations.jsonl
  → 过滤最近 14 天
  → 按 session 分组 → 计算合规率 → 阈值判断
  → 输出 additionalContext（或静默）

Step 4: stop-doc-template-check.js (Stop)
  → 读取 transcript 尾部
  → 检测文档生成操作 → 检测合规标记
  → 输出 systemMessage（或静默）

Step 5: Claude 收到上下文
  → L3 additionalContext → 会话开始就知道"我这周漏了好几次"
  → L2 systemMessage → 刚漏了 → 下一轮立刻自纠
```

**闭环特征**: observe.sh 记录 → L3 分析历史 → 指导当前会话 → L2 验证当前行为 → observe.sh 继续记录。每次漏检都会降低后续会话的合规率，触发更强的提醒。

## 五、扩展指南：如何添加新规则

### 插件式结构

`session-start-compliance-audit.js` 的 `RULES` 对象采用插槽设计——加一行配置就能审计新规则：

```javascript
const RULES = {
  'doc-template-check': {
    name: 'Word 模板确认',
    memoryRef: 'word_templates_choice.md',
    description: '生成 Word/PPT/Excel 文档前应先询问模板偏好',
    triggerPatterns: [
      /\.save\(\s*["'][^"']*\.docx["']/,   // 文档保存
      /Packer\.toBuffer/,                   // docx-js
      // ... 添加新触发模式
    ],
    compliancePatterns: [
      /用哪套模板/,                          // 合规标记
      /Kami.*(风格|样式)/,
      // ... 添加新合规模式
    ],
  },

  // 示例：添加一个"代码审查"规则
  // 'code-review-after-edit': {
  //   name: '代码审查要求',
  //   memoryRef: 'code-review.md',
  //   description: '修改代码后应立即使用 code-reviewer agent',
  //   triggerPatterns: [/* 匹配 Edit/Write 工具调用 */],
  //   compliancePatterns: [/* 匹配 code-review 或 code-reviewer 调用 */],
  // },
};
```

### 添加步骤

1. 在 `RULES` 对象中添加新 key
2. 定义 `triggerPatterns`（正则数组，匹配"做了这个操作"）
3. 定义 `compliancePatterns`（正则数组，匹配"按规定做了"）
4. L2 同步更新：在 `stop-doc-template-check.js` 中添加对应的 `DOC_GEN_PATTERNS` 和 `TEMPLATE_DISCUSSED_PATTERNS`
5. L1 同步更新：在对应 skill 的 SKILL.md 中添加 PRE-GENERATION CHECK

**无需修改**: analyzeRule 函数、阈值逻辑、Hook 注册——全部复用现有基础设施。

## 七、设计决策记录

### 为什么不用纯 LLM 检查

- **成本**: 每次 Stop 都调 LLM → 每次响应增加 500-1000 token。95% 的响应无文档生成 → 全部浪费。
- **延迟**: LLM 调用需要 1-3 秒。Stop hook 应在 10ms 内完成。
- **可靠性**: LLM 可能产生幻觉或误判。正则匹配是确定性行为。

### 为什么两阶段（bash grep + LLM audit）

- 阶段 1 (grep): 95% 的场景在这里结束，零 token。
- 阶段 2 (模式匹配): 剩下 5% 中，大部分是合规通过的，也零 token。
- 只有真正违规时（估计 <1%）才注入 systemMessage（~50 token）。

这个设计将违规检测的边际成本压到了接近零。

### 为什么 systemMessage 而不是 block

- 非破坏性设计哲学：提醒 Claude 自纠，而不是硬阻断。
- 如果误报（比如模板讨论确实发生了但在 scan 窗口外），用户不会被打断。
- block 是 final resort——留给真正的危险操作（如 `rm -rf /`）。

### 为什么阈值设为 85%

- 足够低：不会因为一两次误报就触发全局告警
- 足够高：如果合规率掉到 40%（如首次测试数据），一定触发告警
- 可调：修改 `COMPLIANCE_WARN_THRESHOLD` 常量即可

## 八、缺陷与局限

| 局限 | 影响 | 缓解措施 |
|------|------|----------|
| **跨 turn 匹配** | 前几轮讨论了模板、当前轮生成文档 → L2 可能误报 | systemMessage 非阻断，Claude 可判断为误报 |
| **依赖 transcript 可读性** | 如果 Claude 的 tool output 被截断或编码异常，grep 可能失效 | 多个 pattern 覆盖不同表示形式 |
| **无 real-time 拦截** | L2 在 Stop 时触发，不是 PreToolUse。意味着"生成错了再改"，不是"生成前阻止" | L1 预防层补充；L2 作为安全网 |
| **observations.jsonl 依赖** | L3 需要历史数据才能生效，新部署时无数据 | 静默降级——无数据时不触发告警 |
| **硬编码路径** | session-start-compliance-audit.js 只搜索 `~/ECC/.claude/homunculus` 和 `~/.claude/homunculus` | 如需支持更多路径，修改 `findObservationFiles()` |
| **单规则测试** | 当前只测试了 `doc-template-check` 一条规则 | 架构已支持多规则插槽，等新规则加入时自然覆盖 |
